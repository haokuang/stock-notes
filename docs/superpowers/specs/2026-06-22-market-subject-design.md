# A股大盘研究标的设计

## 背景与目标

当前产品以自选股作为笔记、文档、AI 会话和研究报告的统一归属对象，但用户还需要记录和讨论整个 A 股市场，而不是把内容错误地挂到某一只股票或某一个指数上。

本功能增加一个固定且唯一的「A股大盘」研究标的。用户可在添加自选股页面主动添加它，并围绕它写观点、上传文档、查看历史内容和咨询 AI。它不代表任何具体指数，不提供个股行情、持仓、止损或个股每日简评能力。

## 范围

### 包含

- 每个用户可添加一个固定的「A股大盘」研究标的。
- 大盘标的复用现有自选列表、笔记、文档、AI 会话和 AI 报告链路。
- 首页、资料库、搜索、笔记编辑和详情页能识别并展示大盘标的。
- AI 能识别市场研究模式，并按市场维度组织回答。
- 个股专属的后端操作对大盘标的进行类型保护。

### 不包含

- 上证指数、深证成指、创业板指等具体指数的添加和行情同步。
- 大盘实时行情、K 线、自动每日简评或市场数据采集新链路。
- 多个自定义市场主题。
- 修改现有删除语义；删除标的时仍按当前外键规则级联删除其关联内容。

## 核心模型

在现有 `stocks` 表新增字段：

```text
subject_type: "stock" | "market"
```

- 字段非空，默认值为 `stock`，因此历史数据和旧写入路径保持兼容。
- 固定大盘记录使用：
  - `subject_type = market`
  - `code = MARKET_A_SHARE`
  - `name = A股大盘`
  - `industry = null`
  - `status = watching`
- 现有 `(user_id, code)` 唯一索引保证每个用户最多只有一个大盘标的。
- 笔记、AI 线程、消息和报告继续通过现有 `stock_id` 外键关联，不新增平行主题表。

数据库迁移新增 `subject_type` 列及约束，只允许 `stock`、`market` 两种值。Drizzle schema 同步增加该字段。

## 后端设计

### 创建大盘标的

新增：

```http
POST /api/stocks/market
```

请求不接收业务字段。后端统一写入固定代码、名称和类型，返回与现有创建自选股接口一致的数据信封。成功状态为 HTTP 200。

创建流程：

1. 按当前用户和固定代码查询是否存在。
2. 已存在时返回冲突提示“市场大盘已在自选中”。
3. 不存在时插入固定记录并返回。
4. 不调用 Tushare，不触发行情刷新。
5. 并发写入触发唯一索引时，转换为相同的冲突提示。

### 列表与详情

现有股票列表、详情和搜索响应增加 `subject_type`。默认按当前排序规则展示，不为大盘引入隐式置顶；用户仍可沿用现有排序字段。

### 个股能力保护

以下后端入口在执行前读取标的类型；若为 `market`，返回 HTTP 400 和清晰提示“大盘标的不支持此操作”：

- 刷新价格与刷新状态
- 买入、卖出与止损信息
- 行情历史和每日同步
- 个股每日简评生成与查询

批量行情同步应跳过 `market` 标的，不能让一个大盘记录导致整批同步失败。

### 笔记与 AI

笔记服务继续接受大盘记录的 `stock_id`，并沿用现有所有权校验、查询和汇总逻辑。

AI 侧将标的身份从 `{ code, name }` 扩展为 `{ code, name, subjectType }`。当 `subjectType` 为 `market` 时：

- 系统上下文声明当前研究对象为整个 A 股市场，而非公司或具体指数。
- 回答优先考虑指数表现、市场宽度、成交额、行业轮动、资金流向、风险偏好和市场情绪。
- 不强行输出公司基本面、个股估值、买卖价格或止损价。
- 仍可使用用户在该标的下保存的笔记、文档和已有 AI 报告。
- 新闻检索以“A股市场”和用户问题为语义，不把固定内部代码当成证券代码搜索。
- 个股价格历史和个股画像工具对大盘模式不返回伪造数据；编排器应选择适用于市场研究的上下文和搜索能力。

## 前端设计

### 添加自选页

在股票搜索区上方增加固定的「A股大盘」卡片：

- 使用 `@/components/ui/card` 和 `@/components/ui/button`。
- 显示名称「A股大盘」、标签「市场研究」和说明“记录市场观点，与 AI 讨论指数、行业轮动、资金与情绪”。
- 未添加时显示“添加大盘”；点击调用 `POST /api/stocks/market`。
- 已添加时显示“已添加”并禁用。
- 原有股票搜索、添加及普通 A 股限制保持不变，提示文案明确“下方搜索仅支持沪深北已上市 A 股普通股票”。

### 自选列表、搜索与资料库

- 前端标的类型增加 `subject_type`。
- 大盘项显示“大盘”或“市场研究”标签，不展示行业标签。
- 无价格时使用现有空态，不拼装虚假涨跌数据。
- “全部股票”等包含大盘的筛选文案调整为“全部标的”或“全部研究对象”。
- 点击大盘项仍进入统一详情路由 `/pages/stock/index?stock_id=...`。

### 大盘详情页

详情页根据 `subject_type` 进入市场模式。首次加载先取得标的详情，再决定后续请求，避免并行触发不适用接口。

市场模式只加载：

- 该标的下的笔记和文档
- 观点统计与方向分布
- AI 报告列表

市场模式只展示：

- 「A股大盘」标题和“市场研究”标签
- 问 AI
- 新增观点
- 上传文档
- 历史笔记、文档、观点分布与 AI 报告

市场模式隐藏：

- 当前价、涨跌幅、OHLCV 和价格时间
- 刷新价格
- 买入、卖出、持仓和止损
- 个股每日简评
- 以入场价、目标价、止损价为核心的价格统计卡

普通股票详情行为保持原样。

### 笔记编辑

大盘标的出现在现有标的选择器中。选择后提交逻辑不变，仍发送对应 `stock_id`。涉及“股票”的占位文案改为“自选标的”或“研究对象”，但个股特有价格字段不因本功能新增市场专属逻辑；大盘模式下应隐藏入场价、目标价、止损价输入，方向仍可使用看多、看空、中性表达市场观点。

## 数据流

```text
添加自选页
  -> POST /api/stocks/market
  -> stocks(subject_type=market, code=MARKET_A_SHARE)
  -> 首页 / 搜索 / 资料库列表
  -> 大盘详情
       -> 笔记与文档
       -> AI 线程、消息与报告
       -> 类型感知的市场研究上下文
```

## 错误与兼容性

- 迁移默认值确保所有历史股票自动视为 `stock`。
- 重复添加使用 HTTP 409，不产生重复记录。
- 大盘创建不依赖行情供应商，行情服务异常不影响添加。
- 前端隐藏与后端拒绝同时存在，防止旧客户端或手工请求误触个股操作。
- API 响应继续使用现有 `{ data: ... }` 信封。
- 不修改全局 `/api` 前缀配置，不在 Controller 装饰器中重复加入 `api`。
- 不新增图片、视频、第三方依赖或外部静态资源。

## 测试与验收

### 自动化测试

- 迁移测试：新列、默认值、允许值约束存在，历史记录读取为 `stock`。
- 服务测试：成功创建固定大盘、重复添加冲突、并发唯一冲突映射正确、不调用行情刷新。
- 类型保护测试：大盘不能刷新、买卖、止损、同步或生成个股简评；普通股票不受影响。
- 批量同步测试：跳过大盘并继续处理普通股票。
- 前端逻辑测试：添加卡片状态、详情请求分流、市场模式功能可见性。
- AI 上下文测试：市场标的进入市场提示，不生成个股身份和价格假设。
- 回归执行：相关单测、`pnpm validate`、`pnpm test:prelaunch`。

### 手工验收

1. 新用户进入添加页，能添加一次「A股大盘」。
2. 再次进入添加页时显示“已添加”，重复请求得到明确提示。
3. 首页、搜索和资料库能识别大盘标签并进入详情。
4. 大盘详情不存在价格、交易、止损和个股简评入口。
5. 能为大盘新增观点、上传文档并在详情与资料库中找到。
6. 能创建大盘 AI 会话，AI 按市场维度回答并引用该标的下的用户资料。
7. 删除大盘后可以重新添加；删除语义与现有自选标的一致。
8. 普通股票的添加、详情、行情、交易、简评、笔记和 AI 功能保持正常。

## 成功标准

用户无需借用某只股票或某个指数，即可建立唯一的「A股大盘」研究空间，并稳定完成记录观点、沉淀资料和咨询 AI 三条核心路径；同时，大盘记录不会进入任何个股行情或交易逻辑。
